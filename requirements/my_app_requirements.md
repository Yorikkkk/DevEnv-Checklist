# Специфікація вимог: Чекліст налаштування середовища розробки (DevEnv Checklist)

## Призначення системи та опис продукту
Вебзастосунок «Чекліст налаштування середовища розробки» призначений для автоматизації та систематизації процесу підготовки робочого простору інженера-програмніста. Застосунок розв’язує проблему неефективного використання часу під час онбордингу або старту нових проєктів, коли розробники змушені вручну шукати інструкції та команди для встановлення потрібних утиліт, що часто призводить до інфраструктурних помилок через людський фактор. 

Цільовою аудиторією (користувачами) є починаючі розробники та студенти, яким потрібні чіткі покрокові інструкції, а також досвідчені інженери, що часто перемикаються між різними технологічними стеками. Застосунок має працювати у браузері як статичний клієнтський вебзастосунок, не вимагати backend і бути доступним за публічним посиланням GitHub Pages.

## Межі системи (Scope)
Межі системи визначені її функціонуванням виключно на стороні клієнта (SPA) без використання серверної частини:
* **У межах системи (In-Scope):** реалізовано вибір базових стеків (Frontend, Node.js, Python), інтерактивний трекінг прогресу, пошук та фільтрацію інструментів, а також можливість локального додавання кастомних кроків користувача із автоматичним збереженням стану в `localStorage`.
* **Поза межами системи (Out-of-Scope):** залишаються синхронізація даних між різними пристроями, автоматичне виконання інсталяційних скриптів на реальному ПК користувача (з міркувань безпеки браузерного середовища) та реєстрація чи авторизація користувачів.

## Основні сутності
| Сутність | Опис |
| :--- | :--- |
| Стек розробки | Технологічний напрямок (наприклад, Frontend, Node.js, Python) із назвою, описом та переліком необхідних для встановлення інструментів. |
| Категорія | Тип елемента в чеклісті: Софт, CLI Команда або Розширення. |
| Крок чекліста | Конкретний пункт налаштування, що містить назву інструменту, коротку інструкцію або команду для встановлення. |
| Кастомний крок | Унікальний елемент чекліста, створений безпосередньо користувачем через форму, який додається до поточного стеку. |
| Прогрес | Демонстраційний стан, який у відсотках показує співвідношення виконаних кроків до їх загальної кількості. |

## Функціональні вимоги
| ID | Вимога |
| :--- | :--- |
| FR-01 | Система має відображати список доступних стеків розробки та відповідні їм картки інструментів. |
| FR-02 | Користувач має мати можливість відмічати кроки як виконані або невиконані за допомогою інтерактивних чекбоксів. |
| FR-03 | Користувач має мати можливість шукати інструменти в поточному чеклісті за текстовим запитом. |
| FR-04 | Користувач має мати можливість фільтрувати каталог кроків за категоріями: усе, софт, CLI команди, розширення. |
| FR-05 | Користувач має мати можливість додавати власні кастомні кроки до обраного стеку через спеціальну форму. |
| FR-06 | Form додавання нового кроку має обов'язково валідувати введені дані на порожнечу та довжину рядка (не менше 3 символів). |
| FR-07 | Користувач має мати можливість виконати повне скидання прогресу (Reset Flow) та видалення створених елементів. |
| FR-08 | Система має відображати лічильники загальних, відмічених та знайдених за фільтрами інструментів. |
| FR-09 | Користувач має мати можливість перемикати візуальну тему інтерфейсу (світла / темна). |
| FR-10 | Система має коректно обробляти ситуацію, коли за обраними фільтрами або пошуком нічого не знайдено. |

## Бізнес-правила
| ID | Правило |
| :--- | :--- |
| BR-01 | Елемент чекліста, який має технічну залежність від іншого інструменту, не можна відмітити, доки не виконано базовий крок (наприклад, встановлення Vite заблоковано до встановлення Node.js). |
| BR-02 | Будь-які зміни стану (відмічені чекбокси, додані кастомні кроки, обрана тема) мають автоматично записувалися в `localStorage`. |
| BR-03 | Лічильники прогресу, відсоткова шкала та стан елементів мають оновлюватися миттєво після кожної дії користувача. |
| BR-04 | Якщо результатів пошуку або фільтрації немає, користувач має бачити зрозуміле повідомлення про відсутність результатів. |

## Нефункціональні вимоги
| ID | Вимога |
| :--- | :--- |
| NFR-01 | Застосунок має відкриватися у сучасних браузерах (Chrome, Firefox, Edge) без додаткового встановлення серверної частини. |
| NFR-02 | Основні дії користувача (пошук, фільтрація, перемикання стеків) мають виконуватися без помітної затримки інтерфейсу (до 200 мс). |
| NFR-03 | Інтерфейс має бути адаптивним, виконаний українською мовою та коректно відображатися на екранах моніторів, ноутбуків і смартфонів. |
| NFR-04 | Застосунок не має вимагати введення реальних паролів, токенів, ліцензійних ключів або інших чутливих даних. |

## Критерії приймання
| ID | Критерій приймання |
| :--- | :--- |
| AC-01 | Після відкриття сторінки користувач бачить назву застосунку, перелік доступних стеків розробки, лічильники, panel пошуку та фільтри. |
| AC-02 | Після введення текстового запиту список інструментів змінюється відповідно до запиту, а лічильник знайдених елементів оновлюється. |
| AC-03 | Після вибору категорії (наприклад, "CLI Команда") список елементів змінюється в реальному часі без перезавантаження сторінки. |
| AC-04 | Після додавання кастомного кроку через форму лічильник загальної кількості завдань збільшується, а елемент з'являється у списку. |
| AC-05 | Після натискання кнопки "Скинути прогрес" усі чекбокси знімаються, шкала прогресу стає 0%, а кастомні кроки повністю видаляються з пам'яті. |
